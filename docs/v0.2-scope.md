**Branch:** `v0.2`
**Status:** in progress

---

## In scope

### Functional goals

- Create tasks from the web UI or from OpenClaw (Assistant mode).
- Route each task to an execution backend via deterministic rules.
- Execute tasks with Claude Code or Codex.
- Show progress, logs, artifacts, approvals, and final outcome in Niwa.
- Resume, cancel, or fall back — all auditable.
- Run on Mac mini (full local) and VPS 4–8 GB (control plane + remote runners).

### Two operating modes

| Mode | What it means |
|------|---------------|
| **Core mode** | Niwa standalone. Web UI, task execution, MCP tools, observability. No OpenClaw required. |
| **Assistant mode** | Niwa + OpenClaw. OpenClaw provides Telegram and conversational layer. Niwa remains system of record. |

### New subsystems (v0.2 additions)

| Subsystem | Purpose |
|-----------|---------|
| `backend_profiles` | Registered execution backends (Claude Code, Codex) with capabilities and config. |
| `routing_rules` / `routing_decisions` | Deterministic router: evaluates rules, picks backend, records decision. |
| `backend_runs` / `backend_run_events` | Individual execution attempts with status, heartbeat, outcome, and audit chain. |
| `approvals` | Gate dangerous or expensive operations behind human approval. |
| `artifacts` | Track files produced by backend runs. |
| `project_capability_profiles` | Per-project security policy: filesystem scope, shell access, network, secrets. |
| `secret_bindings` | Bind secrets to specific projects and backends. |
| `assistant_turn` | Unified conversational endpoint used by both web chat and OpenClaw. |
| MCP contract v0.2 | Curated tool surface for OpenClaw (not raw CRUD). |
| Install `--quick` | Streamlined installer with core/assistant mode selection. |

### Bugs to fix (mandatory)

1. Remove the false multi-model illusion (everything ending in `claude -p`).
2. Remove `--dangerously-skip-permissions` from the default path.
3. Deprecate `assigned_to_claude` as execution trigger; replace with `backend_profiles` routing.
4. Normalize `waiting_input` vs `revision` — `waiting_input` is canonical for "needs human answer".
5. Make `_pipeline_status()` count `waiting_input` as active.
6. Move terminal out of default/quick install path.
7. Standardize OpenClaw on `streamable-http` with real smoke test post-registration.

## Out of scope (not in v0.2)

- Desktop app.
- Telegram bot built into Niwa (that's OpenClaw's domain).
- Gemini / Ollama as execution backends.
- LLM-based routing (router is deterministic, rule-based).
- Multi-user / multi-tenant.
- Autonomous loops that create tasks without human trigger.
- Real cost billing / invoicing.
- Full rewrite to a web framework (backend stays stdlib Python).

## Definition of Done (v0.2 release)

1. Core mode works without OpenClaw.
2. Assistant mode installs/registers OpenClaw and leaves Telegram/chat operational.
3. A message from OpenClaw enters via `assistant_turn`.
4. Claude Code and Codex execute real tasks end-to-end.
5. Each execution recorded in `backend_runs` with timeline, heartbeat, and outcome.
6. Fallback and resume create new linked runs.
7. Approval blocks at least one real task before execution.
8. Niwa UI shows routing, runs, approvals, and artifacts.
9. Quick install completes on Mac and Linux in under 10 minutes.
10. VPS 4–8 GB supports control plane + remote runners; Mac mini supports local execution too.
